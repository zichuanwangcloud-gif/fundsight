# 主线 C：数据可靠性收口 - 抓取失败告警通道（M10C-alert-channel）

> 代码名：`M10C-alert-channel`　｜　分支：`feat/m10c-alert-channel`
> 前置：M9-a/b/c 已落地（task_run 可观测 + 系统状态页 + 净值断点检测）。

## 1. 概述

M9-a/b/c 的 `task_run` pass/fail 只是被动展示在状态页，缺主动闭环。补：失败重试 +
连续失败超阈值时主动给受影响用户推 notification。核心价值：抓取层出问题时用户能第一时间
知道，而不是自己打开状态页才发现。目标用户：持仓用户。

## 2. 需求与用户故事

| # | 需求 | 验收标准 |
|---|---|---|
| C1 | 失败重试 | `task_run` fail 时按策略重试 N 次（默认 2 次，间隔递增），重试结果落新 `task_run` 行 |
| C2 | 连续失败告警 | 同一任务连续失败超阈值（默认 3 次）时，给持仓受影响 user 推 `kind=sync_alert` 通知 |
| C3 | 告警去重 | 同 user + 任务名未读的 `sync_alert` 去重，不重复刷屏 |
| C4 | 状态页告警视图 | admin 页加"告警"区，展示未恢复的失败任务 + 受影响基金 |

## 3. 功能验收清单

- [ ] 抓取失败触发重试，重试成功记 ok、仍失败记 fail（C1）
- [ ] 连续 3 次 fail 推 `sync_alert` 通知给持仓 user（C2）
- [ ] 同 user+任务未读告警不重复推送（C3）
- [ ] admin 页渲染告警区，失败任务标红、列受影响基金（C4）
- [ ] 新增测试覆盖：重试成功/失败 / 告警推送 / 去重 / 状态页查询

## 4. 技术约束

- **数据模型**：**复用 `notification` 表**（M9-d 已建），`kind='sync_alert'`，
  `message` 含任务名 + 受影响基金。**不加新表** → 不碰 `db.py`。
- **重试**：`backend/scheduler.py` 的 `_safe_*` 包装层加可选 `retries` 参数，
  `_record_run` 失败时调度重试（新线程，延迟递增）。
- **告警推送**：仿 M9-d `_push_nav_gap_notifications`，新增 `_push_sync_alerts(task_name)`，
  查连续失败计数 + 持仓受影响 user，写 `notification`。
- **daemon**：`start_alert_dispatcher(interval_hours=6)` 巡检连续失败任务并推送。
  `app.py main()` 追加一行调用（落在 `start_session_purge` 之后，`# M10C` 注释）。
- **API**：`backend/api/sync_status.py` 加 `GET /api/admin/sync-alerts`（未恢复失败任务 +
  受影响基金）。`sync_status` 已注册，新路由项加在模块 `ROUTES`，不改 `__init__.py`。
- **前端**：`admin.js` 加告警区渲染，`admin.css` 配色沿用红涨绿跌 + 主色。
- **零依赖**。

## 5. 排除项

- 手机/Web Push 推送——仍 Out of Scope（C 只做应用内 notification）。
- 抓取层换源 / 多源容灾——是数据源层改造，另立项。
- 告警规则自定义编辑界面——本主线只做默认阈值，规则编辑留后续。
