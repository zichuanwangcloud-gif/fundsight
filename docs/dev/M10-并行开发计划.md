## 为什么能并行

项目架构已为并行扩展预留扩展点（见 `backend/api/__init__.py`）：

- **路由注册是追加式**：每个 API 模块导出 `ROUTES`，在 `__init__.py` 锚点追加一行汇总到
  `ALL_ROUTES`。新模块/新路由项互不冲突。
- **scheduler daemon 是独立函数**：每个 `start_*` 是独立线程，`app.py main()` 追加调用即可。
- **数据模型追加式建表**：`init_db` 用 `CREATE TABLE IF NOT EXISTS`，新表追加到 schema 末尾。

三条主线按"模块边界"天然分离，文件集高度不相交，可同时在三个 worktree 并行开发。

## 并行边界约定（三条主线共同遵守）

1. **`backend/api/__init__.py`**：各模块已注册，新增路由项在自己模块的 `ROUTES` 列表里加，
   不改 `__init__.py` 注册行。
2. **`backend/app.py main()`**：追加 daemon 调用一律加在 `start_session_purge(...)` 行之后，
   各自一行，带 `# M10X` 注释标记来源。
3. **`backend/models/db.py` schema**：新增表追加在 `notification` 表 CREATE 语句之后，
   用 `-- M10X: <表名>` 注释块包裹，不修改既有表。
4. **`backend/api/_router.py`**：不改 `dispatch` / `match` 函数体，新中间件以独立函数形式
   添加，由 `app.py` 在分发前调用。
5. **测试隔离**：各主线新测试文件独立，不修改既有 `tests/test_*.py`（既有测试套件 204 通过，
   三主线合并后再统一跑一次）。

## 开发顺序建议

三条主线无依赖，可同时启动。建议各开一个 worktree：

```bash
git worktree add ../fundsight-A  -b feat/m10a-returns-deep   main
git worktree add ../fundsight-B  -b feat/m10b-auth-hardening main
git worktree add ../fundsight-C  -b feat/m10c-alert-channel  main
```

每个 worktree 完成后自测（`bash scripts/test.sh`）→ 提交 → 开 PR → review → 合并。
合并顺序无要求（文件集不相交），但建议 A 先合并（最自包含），B/C 后合并时注意
`app.py main()` 的追加行是否已落在 `start_session_purge` 之后。

## 红线/合规检查（三主线通用）

- 收益分析（A）刻意止于点状/曲线，**不画 K 线、不落 daily 快照**，守住"走势历史图"Out-of-Scope。
- 多用户（B）限流阈值宽松（自用私享级），不引入对外正式运营机制。
- 告警（C）复用站内 notification，不做手机/Web Push（守住"不做推送"红线细化后的边界）。
- 三主线均不得引入第三方依赖（零依赖原则）。
